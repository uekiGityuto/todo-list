# Issue 作成

## テンプレート

`.github/ISSUE_TEMPLATE/task.yml` のフォーマットに従って issue を作成する。

## ラベル

作成時に以下のいずれかのラベルを必ず付ける。

`ready` の issue は自動パイプラインで AI エージェントに渡され、ユーザーへの質問なしに PR まで作成される前提。そのため `ready` には、エージェントが自律的に実装完了できるだけの情報が必要。

| ラベル | 条件 |
|---|---|
| `ready` | AI エージェントがユーザーに質問せずに実装・PR 作成できるレベルで、要件・参照資料・確認方法が揃っている |
| `needs-refinement` | 実装時にユーザーへの質問が必要になりうる（要件が曖昧、設計判断が必要、情報が不足等） |

## ラベルの判断フロー

### 仕様が明確な場合（仕様を会話で決めた、または単純なタスクで仕様が自明）

テンプレートに従って要件を書き、起票前にユーザーに確認する:

1. この仕様で良いか → `ready` で起票
2. 仕様を直すか → ユーザーと相談して修正し、再度確認
3. `needs-refinement` で登録するか → どこが曖昧か・何を追加で検討すべきかをユーザーに確認してから起票

### それ以外の場合（仕様について特に話していない等）

`needs-refinement` で作成する。起票前にユーザーに内容を確認する。

### 本文の書き方

- 余計なことは書かず、ユーザーに書いてと言われたことだけ書く
- 確定していないことを断定的に書かない。迷うならユーザーが言った通りに書く
- 作業時の注意点があれば書いてよいが、作業する人が惑わされないようにする

## コマンド例

```bash
gh issue create --title "タイトル" --body "本文" --label "ready"
```
