# E2E テストカバレッジルール

## 何を E2E でテストするか

- ブラウザで開いて操作したときに壊れやすい、主要画面の表示確認
- Web と API の結合が絡む代表的な CRUD
- アプリ固有で重要な導線や操作

## 何を E2E でテストしないか

- pure logic の検証
- API 単体の振る舞い
- 見た目や状態差分の確認が主目的のもの
- Storybook や Vitest で十分に担保できるもの

## 使い分け

| 対象 | テスト種別 | 備考 |
|------|-----------|------|
| pure な関数・フック | 単体テスト | Vitest |
| API の入出力・ユーザー隔離 | 結合テスト | API テスト |
| UI の見た目や状態差分 | Storybook | Story / VRT |
| 実ブラウザでの画面表示・導線・結合確認 | E2E | Playwright |

## 優先順位

1. 認証済み主要ページが開けること
2. 代表的な CRUD が 1 つ通ること
3. アプリ固有の重要操作が 1 つ通ること

## 原則不要

- 画面を開かずに検証できるもの
- 同じ価値をより安定に lower-level test で担保できるもの
- 初期開発段階で保守コストの方が高い、細かすぎる操作網羅
