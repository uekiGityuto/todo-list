# API テストカバレッジルール

## 基本方針

1. **単体テストを原則書く**
2. API・DB 依存で単体テストが書きづらい場合は **結合テストで代替可**
3. 各 feature には **結合テスト（`app.request()` + テスト DB）を必ず書く**

## テスト不要の判断基準

以下の **すべて** に該当するファイルはテスト不要:

- 条件分岐・データ変換などのロジックがない
- ライブラリの初期化や設定値のエクスポートのみ

例: `shared/lib/prisma.ts`（Prisma client のエクスポート）、`shared/auth/env.ts`（環境変数の定義）

## テスト種別の選び方

| 対象 | テスト種別 | 備考 |
|------|-----------|------|
| `features/*/route.ts` | 結合テスト | `app.request()` + テスト DB |
| `features/*/service.ts` | 結合テストで網羅 | route 経由のテストでカバー |
| `features/*/domain.ts` | 単体テスト | pure logic、DB 非依存 |
| `shared/` 内のロジック | 単体テスト優先 | 単体テストが書きづらい場合は feature の結合テストでカバーでも可 |

## 規約

- テストファイルは `__tests__/` ディレクトリに `<対象ファイル名>.test.ts` で配置する
- テストランナー: Vitest
